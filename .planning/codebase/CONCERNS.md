# Codebase Concerns

**Analysis Date:** 2026-04-12

## Tech Debt

**Unbuilt application - scaffold only:**
- Issue: The entire codebase is the default `create-next-app` scaffold. No resume content, components, or application logic exists.
- Files: `src/app/page.tsx`, `src/app/layout.tsx`
- Impact: The application serves no purpose in its current state. All delivery value is blocked.
- Fix approach: Replace `src/app/page.tsx` content with actual resume UI. Update metadata in `src/app/layout.tsx`.

**Boilerplate metadata never updated:**
- Issue: `layout.tsx` exports `title: "Create Next App"` and `description: "Generated by create next app"` — placeholder values from project initialization.
- Files: `src/app/layout.tsx` (lines 15–18)
- Impact: Poor SEO, incorrect browser tab title, unprofessional appearance in search results and link previews.
- Fix approach: Set accurate `title` and `description` in the `metadata` export in `src/app/layout.tsx`.

**Unused default assets committed:**
- Issue: `public/` contains five default Next.js SVGs (`file.svg`, `globe.svg`, `next.svg`, `vercel.svg`, `window.svg`) from the scaffold. None are referenced by application logic except `next.svg` in the placeholder page.
- Files: `public/file.svg`, `public/globe.svg`, `public/next.svg`, `public/vercel.svg`, `public/window.svg`
- Impact: Unnecessary files in the repository; will cause confusion about what assets are intentional.
- Fix approach: Delete all five files once the placeholder `page.tsx` is replaced with real content.

**Placeholder page references external Vercel/Next.js marketing URLs:**
- Issue: `src/app/page.tsx` contains hard-coded `href` links to `vercel.com` and `nextjs.org` marketing pages. These are scaffold remnants, not application features.
- Files: `src/app/page.tsx` (lines 22–35, 38–61)
- Impact: If the page is deployed without being replaced, users see a framework advertisement rather than a resume.
- Fix approach: Delete all content in `src/app/page.tsx` and build resume sections.

**Font CSS variable defined but overridden in body:**
- Issue: `globals.css` sets `--font-sans` and `--font-mono` via `@theme inline` to point at the Geist font CSS variables, but the `body` rule then overrides `font-family` with `Arial, Helvetica, sans-serif`. This means the Geist font loaded via `next/font/google` in `layout.tsx` is unused on `body` text.
- Files: `src/app/globals.css` (lines 8–13, 24–26), `src/app/layout.tsx` (lines 5–12)
- Impact: Inconsistent typography; Geist font is loaded but not applied to body text, causing a wasted network request and visual inconsistency.
- Fix approach: Remove the `font-family` override from the `body` rule in `globals.css`, or replace the Tailwind `font-sans` utility with the correct variable reference.

**`types/cache-life.d.ts` and `types/validator.ts` are auto-generated files committed to source:**
- Issue: `types/routes.d.ts` and `types/validator.ts` carry the comment "This file is generated automatically by Next.js / Do not edit this file manually". These are build artifacts.
- Files: `types/cache-life.d.ts`, `types/routes.d.ts`, `types/validator.ts`
- Impact: Generated files in source control can fall out of sync with the actual build, create misleading diffs, and cause confusion about what is hand-authored. `next-env.d.ts` is already excluded via `.gitignore` but these `types/` files are not.
- Fix approach: Add `types/routes.d.ts` and `types/validator.ts` to `.gitignore` if they are truly generated. Verify whether `types/cache-life.d.ts` is also generated and should be excluded.

## Security Considerations

**No Content Security Policy (CSP) configured:**
- Risk: Next.js does not configure a CSP by default. The application has no custom headers.
- Files: `next.config.ts`
- Current mitigation: None.
- Recommendations: Add `headers()` config in `next.config.ts` to set `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy` response headers.

## Test Coverage Gaps

**No tests exist:**
- What's not tested: Everything — there are no test files, no test configuration, and no test runner installed.
- Files: Entire `src/` directory
- Risk: Any future code added to the application ships without automated validation.
- Priority: Medium — low urgency while the codebase is scaffold-only, but must be addressed before substantive application logic is added.

## Missing Critical Features

**No resume content:**
- Problem: The application is named "resume" but contains zero resume data, sections, or components. There is no data model, no rendering logic, and no content.
- Blocks: The application cannot be deployed meaningfully or evaluated until actual resume content is implemented.

**No error boundary or 404 page:**
- Problem: No `not-found.tsx` or `error.tsx` exists under `src/app/`. Next.js App Router falls back to a generic error UI.
- Files: `src/app/` (missing `not-found.tsx`, `error.tsx`)
- Risk: Poor user experience and no opportunity to handle errors gracefully.

## Fragile Areas

**`next.config.ts` is empty of actual configuration:**
- Files: `next.config.ts`
- Why fragile: The config object contains only `reactCompiler: true` with a comment placeholder. Any future configuration needed (image domains, headers, rewrites, redirects) has no established home or pattern.
- Safe modification: Add new config keys to the `nextConfig` object following Next.js 16 conventions — read `node_modules/next/dist/docs/` before modifying, as the project `AGENTS.md` warns of breaking API changes.
