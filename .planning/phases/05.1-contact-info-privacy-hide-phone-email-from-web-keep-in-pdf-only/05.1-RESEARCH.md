# Phase 05.1: Contact Info Privacy - Research

**Researched:** 2026-03-06
**Domain:** CSS print/screen media, privacy, static site data handling
**Confidence:** HIGH

## Summary

This phase hides the user's phone number and email from the public web version while preserving them in the PDF/print export. The codebase already has all the infrastructure needed: `react-to-print` triggers `window.print()`, print styles in `src/styles/print.css` with a `.no-print` class pattern, and the contact info is rendered in `Header.tsx` with clearly separated elements per contact type.

The implementation is straightforward CSS-driven visibility toggling. Elements marked with a "print-only" class are hidden on screen but shown in print. The existing `.no-print` pattern (hide in print) is simply inverted to `.print-only` (hide on screen, show in print). No new libraries are needed.

**Primary recommendation:** Use CSS classes to toggle visibility -- `.print-only` elements hidden on screen, visible in `@media print`. Apply to email and phone contact items in Header.tsx. Also remove email/telephone from JSON-LD structured data in page.tsx.

## Architecture Patterns

### Pattern 1: CSS Print-Only Visibility Toggle

**What:** A `.print-only` CSS class that hides elements on screen but reveals them in print, the inverse of the existing `.no-print` class.

**When to use:** Any content that should appear in PDF but not on the website.

**Example:**
```css
/* In globals.css or print.css */
.print-only {
  display: none !important;
}

@media print {
  .print-only {
    display: flex !important; /* or block/inline as needed */
  }
}
```

**Confidence:** HIGH -- This is the standard pattern used across the web. The project already uses `.no-print` (the inverse) successfully.

### Pattern 2: Conditional Wrapper in Header Component

**What:** Wrap email and phone `<a>` elements with the `.print-only` class so they render in the DOM (available for print) but are visually hidden on screen.

**Example:**
```tsx
{/* Email - print only */}
<a
  href={`mailto:${contact.email}`}
  className="print-only group flex items-center gap-2 px-4 py-2 ..."
>
  <MailOutlined className="text-[var(--accent)]" />
  <span className="...">{contact.email}</span>
</a>

{/* Phone - print only */}
{contact.phone && (
  <a
    href={`tel:${contact.phone}`}
    className="print-only group flex items-center gap-2 px-4 py-2 ..."
  >
    <PhoneOutlined className="text-[var(--accent)]" />
    <span className="...">{contact.phone}</span>
  </a>
)}
```

### Pattern 3: Remove from JSON-LD Structured Data

**What:** The JSON-LD in `page.tsx` currently exposes `email` and `telephone` in the Person schema. These must also be removed to prevent search engines from indexing the private contact info.

**Current code (page.tsx lines 36-37):**
```tsx
email: contactInfo.email,
telephone: contactInfo.phone,
```

**Action:** Remove both `email` and `telephone` properties from the JSON-LD `mainEntity`.

### Anti-Patterns to Avoid

- **JavaScript-based hiding (e.g., state toggle before print):** Overly complex. CSS handles this natively and `react-to-print` uses the browser's print API which respects `@media print`.
- **Removing data from `resume.ts`:** The data must stay in the source so it renders in the PDF. Only the visual display changes.
- **Using `visibility: hidden` instead of `display: none`:** `visibility: hidden` still takes up space in the layout, causing gaps on the web version.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Print/screen toggle | JS-based show/hide with state | CSS `.print-only` class | Browser handles `@media print` natively, react-to-print already uses `window.print()` |
| Data obfuscation | Base64 encoding, JS decryption | Simply don't display on screen | Over-engineering; the data is in the static JS bundle regardless, but not showing it in the DOM or structured data is sufficient |

**Key insight:** The email/phone will still exist in the JavaScript bundle (in `resume.ts`). True obfuscation from determined scrapers is impossible with a static site. The goal is practical privacy: not displaying in the visible page, not indexing in structured data, and not being in the HTML source that casual visitors or search engines see.

## Common Pitfalls

### Pitfall 1: Forgetting the JSON-LD Structured Data
**What goes wrong:** Email/phone are hidden visually but still exposed in the `<script type="application/ld+json">` block, making them harvestable by search engines.
**Why it happens:** Developer focuses on visual changes, forgets about the structured data in `page.tsx`.
**How to avoid:** Remove `email` and `telephone` from the JSON-LD Person schema.
**Warning signs:** Searching page source for the email address still finds it in the JSON-LD.

### Pitfall 2: Print Layout Mismatch
**What goes wrong:** Email/phone appear in print but layout looks different from expected (wrong alignment, spacing).
**Why it happens:** The `.print-only` elements go from `display: none` to visible, and print CSS may not fully restore the intended layout.
**How to avoid:** Test with `Ctrl+P` / `Cmd+P` in browser to preview print layout. Ensure the `display` value in the `@media print` rule matches the element's natural display mode (flex for the contact pill layout).
**Warning signs:** Contact pills stacking vertically or having inconsistent spacing in print preview.

### Pitfall 3: Screen Reader Accessibility
**What goes wrong:** `display: none` elements are invisible to screen readers, so a blind user cannot access the email/phone even on the web.
**Why it happens:** `display: none` removes elements from the accessibility tree.
**How to avoid:** This is actually the desired behavior for privacy -- we do not want the info accessible on the web version in any form. If the user later wants screen reader access, use `aria-hidden` + visual-only hiding instead, but for privacy, `display: none` is correct.

### Pitfall 4: `next/head` or Metadata Leaking Contact Info
**What goes wrong:** Contact info might be in OG meta tags or other metadata.
**How to avoid:** Check `layout.tsx` metadata -- verified that the current metadata does not include email/phone. The only leak point is JSON-LD.
