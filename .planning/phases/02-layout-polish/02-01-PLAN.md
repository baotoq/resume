## Context

Phase 1 components (`Header`, `WorkExperience`, `Skills`) are Server Components. Framer Motion's `motion.*` requires `'use client'`. The solution is a thin `AnimateIn` client wrapper that receives Server Component output as `children`. This is the canonical Next.js pattern for mixing animation libraries with Server Components.

UI-SPEC specifies:
- Animation: `opacity: 0, y: 16` → `opacity: 1, y: 0`, duration `0.4s`, easing `easeOut`
- Trigger: `whileInView`, fires once (`viewport={{ once: true }}`)
- Stagger: Header delay `0s`, WorkExperience delay `0.1s`, Skills delay `0.2s`

<action>
Create new file `src/components/AnimateIn.tsx` with the following exact content:

```tsx
'use client'

import { motion } from 'framer-motion'

interface AnimateInProps {
  children: React.ReactNode
  delay?: number
}

export function AnimateIn({ children, delay = 0 }: AnimateInProps) {
  return (
    <motion.div
      initial={{ opacity: 0, y: 16 }}
      whileInView={{ opacity: 1, y: 0 }}
      viewport={{ once: true }}
      transition={{ duration: 0.4, ease: 'easeOut', delay }}
    >
      {children}
    </motion.div>
  )
}
```

Key points:
- `'use client'` directive at the top — required by Framer Motion
- `initial={{ opacity: 0, y: 16 }}` — hidden, 16px below final position (per UI-SPEC)
- `whileInView={{ opacity: 1, y: 0 }}` — animate to visible when entering viewport
- `viewport={{ once: true }}` — fires once only, stays visible on scroll up (per CONTEXT.md D-04)
- `transition={{ duration: 0.4, ease: 'easeOut', delay }}` — 0.4s easeOut (per UI-SPEC)
- `delay` prop defaults to `0` — caller passes `0`, `0.1`, `0.2` for stagger
</action>
